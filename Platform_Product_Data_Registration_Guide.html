핵심 테제

AI 에이전트가 상품을 "발견 → 비교 → 추천 → 결제"하는 에이전틱 커머스 시대에, 상품 데이터를 어디에, 어떤 형식으로, 어떤 품질로 등록하느냐가 곧 매출을 결정합니다. 각 플랫폼은 서로 다른 프로토콜(ACP, UCP, 독자 규격)로 데이터를 수집하며, 데이터 품질 검증 기준도 플랫폼마다 다릅니다. 이 리포트는 판매자가 "한 번 정리한 상품 데이터로 모든 AI 플랫폼에 동시 노출"하기 위해 알아야 할 모든 것을 다룹니다.

1 핵심 구조 이해: Push vs Pull vs Discovery

플랫폼마다 상품 데이터를 수집하는 방식이 근본적으로 다릅니다

3가지 데이터 수집 패러다임

① Push 모델 — 판매자가 직접 데이터 파일을 플랫폼 서버에 "밀어 넣는" 방식 (예: OpenAI SFTP, Google Feed 업로드)
② Pull 모델 — 플랫폼이 판매자 사이트에서 데이터를 "가져가는" 방식 (예: Google 크롤링, Perplexity 웹 수집)
③ Discovery 모델 — AI 에이전트가 표준 엔드포인트를 자동 탐색하는 방식 (예: UCP의 .well-known/ucp)

방법 B: UCP (Universal Commerce Protocol) — AI 에이전트 시대의 새 표준

UCP란?

Google과 Shopify가 2026년 1월 NRF에서 공동 발표한 오픈소스 커머스 프로토콜입니다. 판매자가 자신의 도메인에 /.well-known/ucp 매니페스트를 배포하면, AI 에이전트가 자동으로 "이 사이트에서 뭘 살 수 있는지" 발견합니다. 크롤링이나 피드 제출 없이, 에이전트가 스스로 찾아오는 Discovery 모델입니다.

STEP 2

→
STEP 3

// 예시: https://myshop.co.kr/.well-known/ucp { "version": "1.0", "merchant": { "name": "MyShop Korea", "domain": "myshop.co.kr", "country": "KR" }, "services": [ { "type": "shopping", "capabilities": ["product_search", "checkout", "native_commerce"], "catalog_endpoint": "/api/ucp/catalog", "checkout_endpoint": "/api/ucp/checkout" } ], "payment_handlers": ["google_pay", "shop_pay", "card"], "extensions": { "fulfillment": { "methods": ["delivery", "pickup"] }, "discounts": { "enabled": true } } }
UCP 매니페스트 핵심 필드설명AI 에이전트 활용
services[]지원하는 서비스 목록 (shopping, booking 등)에이전트가 가능한 액션을 파악
capabilities[]각 서비스의 세부 기능 (checkout, search 등)체크아웃 가능 여부 즉시 판단
payment_handlers[]지원 결제 수단 목록사용자 결제 수단과 매칭
catalog_endpoint상품 카탈로그 API 주소실시간 상품 검색 & 조회
extensions{}배송, 할인, 반품 등 확장 기능조건부 추천 로직에 활용
native_commerceAI 내 직접 결제 허용 플래그Google AI Mode에서 즉시 구매 활성화

2025~2026 주요 변경사항

2025년 4월
할부 가격 정책 변경 — installment 속성의 down_payment 하위 속성 필수화, price 필드에 할부가 아닌 일시불 가격 기재

2026년 1월 (NRF)
UCP 공식 발표 — Google + Shopify 공동 개발, 오픈소스 표준. native_commerce 속성 도입으로 AI Mode 내 직접 결제 지원
2026년 진행중
AI 전용 속성 확대 — FAQ, 호환 액세서리, 대체 상품, 대화형 속성(conversational attributes) 등 수십 개 신규 필드 추가

3 OpenAI 에코시스템 상품 등록

ACP (Agentic Commerce Protocol) + ChatGPT Merchant Portal — AI 네이티브 커머스

O

OpenAI ChatGPT Shopping ACP 프로토콜

ChatGPT 내 상품 발견 → 비교 → 결제 (Stripe SPT) — 현재 미국 판매자만 온보딩 가능

⚠ 핵심 차이: "ChatGPT는 웹을 크롤링하지 않습니다"

Google Shopping이 Merchant Center에서 피드를 가져가거나 웹사이트를 크롤링하는 것과 달리, ChatGPT는 판매자가 직접 구조화된 파일을 OpenAI 서버에 전송(Push)하는 모델입니다. 즉, 피드를 보내지 않으면 ChatGPT에 당신의 상품은 존재하지 않습니다.

온보딩 5단계 프로세스

OpenAI가 고유한 보안 SFTP 엔드포인트 + 인증 자격증명 제공
→
STEP 3
샘플 피드 제출
대표 상품 100개로 파싱 + 스키마 검사 테스트

STEP 4

5 플랫폼별 데이터 품질(정확성) 검증 방법

각 플랫폼이 상품 데이터의 품질을 검증하고 거부하는 기준과 방법

핵심 원칙: "AI 에이전트는 부정확한 데이터를 증폭시킵니다"

전통 쇼핑에서 부정확한 가격은 고객이 장바구니에서 발견하지만, AI 에이전트가 잘못된 가격을 "추천"하면 브랜드 신뢰가 대규모로 손상됩니다. 이 때문에 모든 플랫폼이 데이터 품질 검증을 크게 강화하고 있습니다.

● Google Merchant Center

3단계 심각도: Error → Warning → Suggestion

  • 가격 불일치 — 피드 가격과 랜딩 페이지 실제 가격이 다른 경우 (가장 빈번)
  • GTIN 오류 — 잘못되거나 누락된 GTIN → 즉시 비승인
  • 이미지 품질 — 100×100px 미만, 워터마크, 프로모션 텍스트 포함 이미지
  • 필수 필드 누락 — id, title, price, availability 중 하나라도 누락 시 전체 SKU 거부
  • 배송 정확성 — 배송비, 예상 배송일이 실제와 불일치 (2025 강화)
  • 반복 비승인 — 경고(Warning)를 장기간 방치하면 계정 정지로 격상

모니터링: Diagnostics 탭 (실시간), Merchant Center API (자동화)

● OpenAI ChatGPT

샘플 검증 → 전체 배포의 게이트키핑 모델

  • 필수 필드 누락 — product_id, title, price, availability 중 하나라도 없으면 전체 SKU 거부
  • 데이터 타입 오류 — 가격이 소수자 아닌 문자열, 통화 코드가 ISO 4217 미준수
  • 잘못된 URL — image_link, link 필드의 URI가 유효하지 않거나 접근 불가
  • 비고유 상품 ID — 같은 product_id가 여러 행에 중복 사용
  • 마케팅 문구 사용 — "최고의!", "필수 아이템!", "대박 할인!" 등 사실이 아닌 설명
  • 가격 불일치 — 피드와 웹사이트 실제 가격이 다르면 AI가 교차 검증 후 즉시 거부

검증 단계: ① 파싱 검증 → ② 스키마 컴플라이언스 → ③ 크로스 사이트 가격 확인

● Amazon SP-API

Product Type Definitions 기반 동적 스키마 검증

  • 카테고리별 필수 속성 누락 — Product Type마다 다른 필수 필드 (동적 스키마)
  • ASIN 매칭 실패 — 기존 카탈로그 ASIN과 일치하지 않는 상품 데이터
  • 브랜드 레지스트리 — Brand Registry 미등록 브랜드의 자체 브랜드 주장 시 거부
  • 이미지 규격 — 메인 이미지: 순백 배경(RGB 255,255,255), 최소 1000×1000px
  • 금지어 사용 — 제목에 "최저가", "무료배송", HTML 태그 등 포함 시 거부
  • 단위 형식 (2025 변경) — 약어("lb") 대신 전체명("pounds") 사용 필수

검증: SP-API getFeed 응답의 processingStatus + 개별 오류 코드

● Meta Commerce

Commerce Manager Diagnostics 기반

  • 필수 필드 누락 — id, title, description, availability, image_link 중 하나 누락 시 거부
  • 재고 정확성 — quantity 필드가 정수가 아닌 경우, 0 상태 56일 이상 시 자동 숨김
  • 이미지 정책 — 텍스트 오버레이, 행동유도문구 포함 이미지 거부
  • 카테고리 불일치 — fb_product_category와 실제 상품이 불일치
  • 가격/통화 오류 — 가격 형식 오류 또는 지원되지 않는 통화
  • 체크아웃 추가 요건 — Checkout 활성화 시 quantity, checkout URL 등 추가 필드 미제출

모니터링: Commerce Manager 진단 탭 + Catalog Batch API

● Shopify Catalog

AI 자동 정제 + 프로토콜 준수 자동화

  • AI 카테고리 자동 추론 — 상품 정보에서 카테고리를 자동 분류 (수동 매핑 불필요)
  • 속성 자동 추출 — 제목/설명에서 색상, 사이즈, 소재 등 구조화 속성 추출
  • 변형 자동 통합 — 같은 상품의 색상/사이즈 변형을 하나의 product group으로 클러스터링
  • 가격/재고 실시간 동기화 — Shopify Admin과 모든 AI 채널 간 자동 일관성 유지
  • 프로토콜 자동 변환 — ACP(OpenAI), UCP(Google) 각 규격에 맞춰 자동 변환 후 배포

장점: 판매자가 데이터 품질을 직접 관리할 필요 없이 Shopify AI가 대부분 처리

● Perplexity

웹 수집 + 파트너 데이터 교차 검증

  • 웹 크롤링 검증 — 웹 검색 결과와 파트너 데이터를 교차 대조
  • Shopify 연동 품질 — Shopify 인프라를 통해 연동된 데이터는 이미 정제된 상태
  • Merchant Program — 직접 가입 머천트에게 더 완전한 데이터 접근권을 부여하여 정확성 향상
  • 실시간 가격 확인 — Instant Buy 결제 전 실제 가격 실시간 확인

6 프로토콜과의 관계: ACP · UCP · MCP

3대 오픈 프로토콜이 플랫폼 상품 등록과 어떻게 연결되는가

에이전틱 커머스 프로토콜 ↔ 플랫폼 매핑

🔴 ACP (Agentic Commerce Protocol)

주도: OpenAI + Stripe

방식: Push — 판매자가 구조화된 피드를 직접 전송

결제: Stripe SPT (Shared Payment Token)

적용: ChatGPT Shopping, Shopify Agentic Storefronts

판매자 → SFTP/HTTPS → OpenAI 서버 → ChatGPT가 상품 검색·추천

🟡 UCP (Universal Commerce Protocol)

주도: Google + Shopify

방식: Discovery — .well-known/ucp 매니페스트 자동 탐색

결제: Google Pay, Shop Pay 등 다중 지원

적용: Google AI Mode, Gemini, Shopify, 기타 UCP 호환 에이전트

판매자 도메인에 매니페스트 배포 → AI 에이전트가 자동 발견 → 실시간 API 호출

🟣 MCP (Model Context Protocol)

주도: Anthropic

방식: Pull — AI 모델이 직접 데이터 소스에 쿼리

결제: 직접 결제 기능 없음 (Stripe 파트너십으로 보완)

적용: Akeneo MCP Server, 토스페이먼츠 MCP Server, 카카오 PlayMCP

AI 모델 → MCP 서버 → 데이터 소스 직접 쿼리 (브랜드 가이드라인 준수 보장)

비교 항목 ACP UCP MCP
데이터 흐름 방향 판매자 → 플랫폼 (Push) 에이전트 → 판매자 도메인 (Discovery) AI 모델 → 데이터 소스 (Pull)
상품 데이터 형식 JSONL/CSV/Parquet 피드 파일 JSON 매니페스트 + REST API 엔드포인트 MCP 서버가 정의한 스키마 (자유 형식)
데이터 신선도 15분 주기 배치 업데이트 실시간 API 호출 실시간 쿼리
판매자 부담 피드 파일 생성 + 주기적 전송 .well-known/ucp 배포 + API 구현 MCP 서버 구축 or 연동
데이터 품질 관리 플랫폼이 피드 검증 후 거부/승인 에이전트가 API 호출 시 실시간 검증 MCP 서버 내 거버넌스 레이어
결제 통합 Stripe SPT 내장 payment_handlers로 선언 별도 (Stripe 파트너십 등)
오픈소스 ✅ GitHub 공개 ✅ 오픈소스 ✅ 오픈소스
한국 준비도 🔴 낮음 (미국만 온보딩 가능) 🟡 보통 (글로벌, 구현 필요) 🟢 높음 (토스·카카오 이미 도입)

프로토콜이 상품 등록에 미치는 구체적 영향

핵심 이해: "프로토콜 = 상품 데이터의 교통 규칙"

프로토콜은 "어떤 데이터를, 어떤 형식으로, 어떤 방식으로 전달할 것인가"를 정의하는 규칙입니다. 같은 상품 데이터라도 ACP로 보내면 ChatGPT에, UCP로 배포하면 Google AI Mode에, MCP로 연결하면 Anthropic Claude에 노출됩니다.

상황필요한 프로토콜구체적 행동
"ChatGPT에서 우리 상품이 추천되게 하고 싶다" ACP JSONL 피드 파일을 ACP 스펙에 맞춰 작성 → OpenAI Merchant Portal에서 SFTP 전송
"Google AI Mode에서 우리 상품을 직접 결제까지 되게 하고 싶다" UCP 도메인에 .well-known/ucp 매니페스트 배포 + native_commerce=true + checkout API 구현
"모든 AI에 동시에 노출하고 싶다" ACP + UCP Shopify에 상품 등록 → Shopify Catalog가 양쪽 프로토콜로 자동 배포
"AI가 우리 브랜드 가이드라인을 지키면서 상품을 설명하게 하고 싶다" MCP Akeneo 같은 PIM에 MCP 서버 구축 → 브랜드 룰 + 법적 면책 조항 거버넌스 레이어 추가
"Shopify 없이 한국에서 직접 AI 커머스 파이프라인을 만들고 싶다" ACP + UCP + MCP 한국형 Feed Transformer SaaS 구축 — 상품 DB → ACP/UCP/MCP 3중 출력

프로토콜별 데이터 품질 관리 철학

ACP: "게이트키퍼 모델"

플랫폼(OpenAI)이 피드를 수신 후 검증하고, 기준 미달 시 거부(reject)합니다. 판매자는 거부 사유를 받고 수정 후 재전송합니다. 마치 출판사가 원고를 검수하는 것과 같습니다.

→ 데이터 품질 책임: 판매자 + 플랫폼 공동

UCP: "자기 선언 모델"

판매자가 자기 도메인에 매니페스트를 직접 배포하고, AI 에이전트가 이를 신뢰하고 사용합니다. 잘못된 정보를 배포하면 에이전트가 해당 판매자를 신뢰도 저하로 분류합니다.

→ 데이터 품질 책임: 판매자 전적 책임

MCP: "거버넌스 모델"

MCP 서버 내에 거버넌스 레이어를 두어, AI 모델이 쿼리할 때 브랜드 가이드라인, 법적 요건, 가격 정책 등을 강제 적용합니다. Akeneo가 대표적 구현체입니다.

→ 데이터 품질 책임: MCP 서버 (거버넌스 자동화)

ACP 피드 규격은 현재 가장 엄격한 상품 데이터 표준입니다. 이 기준에 맞춰 상품 데이터를 정비하면:
① Google Merchant Center 피드 품질도 자동으로 향상 (필수 필드가 대부분 겹침)
② UCP 매니페스트의 카탈로그 API에도 동일한 정제 데이터 활용 가능
③ MCP 서버 구축 시에도 같은 데이터 모델을 재사용
④ 네이버 쇼핑 인텔리전스, 카카오 PlayMCP 등 국내 플랫폼에도 바로 적용 가능
